# 轻量开发流程

> **执行状态**: 📄 活跃引用文档 — 被 AGENTS.md §4.7 调用，按需读取

> 本文件由 AGENTS.md §4.7 引用，仅在用户触发「立项」时按需读取。

**触发词**：`立项`（去除标点后精确等于这两个字）

**防误触**：
- 消息精确匹配「立项」→ 进入立项流程
- 消息包含「立项」但还有其他内容 → 视为正常对话，不触发

**适用场景**：
- 中小型功能开发（如新增 CLI 命令、模块局部改造）
- 已有明确方向的技术改进（如配置合并、代码清理）
- 不需要 proposal/design/tasks/prompt 全链路的改动

**不适用**：大型跨模块功能、涉及多 agent 接力的复杂项目 → 走 `agent-coding-workflow.md` 五阶段 SOP。

---

## 与 SOP 的边界判断

| 判定条件 | 建议走轻量 | 建议走 SOP |
|----------|-----------|-----------|
| 涉及 > 3 个模块 | ❌ | ✅ |
| 需要多 agent 接力 | ❌ | ✅ |
| 有外部 API/DB schema 变更 | 可 | ✅ |
| 可在一个会话内完成 | ✅ | 可 |
| 改动 < 5 个文件 | ✅ | 可 |
| 新增独立功能（不碰核心架构）| ✅ | 可 |

> Agent 不确定时主动询问用户选择。

## 确认流程

1. 读取 `status.md` 确认当前项目状态
2. 输出「轻量立项」提示：
   ```
   【轻量立项】
   📌 当前状态：[从 status.md 提取]
   💡 请描述要做什么，我来确认方案方向。
   ```
3. 用户描述需求后，进入立项讨论（见下方 ①）
4. 讨论结束后 Agent 输出方案方向摘要
5. 用户确认 `y` → 进入定方案（②）；`n` → 返回第 3 步继续讨论

> **无响应/退出**：Agent 提示后如连续 3 轮无有效需求描述，输出「立项已取消」并退出流程，恢复普通对话模式。用户也可随时说「取消」中止立项。

## 标准动作序列

```
① 立项讨论 ── Agent 通过提问摸清需求，参考 SOP 阶段一的交互模式
    │        （用提问确认、不猜测意图、不明确处必须问）
    │
② 定方案 ── 写入 docs/<方案名>.md（按下方模板）
    │         更新 status.md 轻量方案字段、更新记录表
    │
③ 执行 ── 按方案中的「执行步骤」逐项实施
    │       每步完成后勾选 [ ] → [x]
    │       当前 agent 或后续 agent 接力均可，不固定会话边界
    │
④ 标注 + 归档 ── 执行完毕后更新状态，移入 docs/archived/
```

### ① 立项讨论

参考 `agent-coding-workflow.md` 阶段一的交互模式，但省略 T-01~T-05 技术需求确认（那是 SOP 阶段二的输入条件）：

**核心原则**：
- 用提问方式帮助用户收敛需求
- **绝不猜测用户意图**——所有不明确处必须提问
- 用户没有技术背景时，用大白话解释

**必问项**（Agent 在讨论中主动确认）：
- 要解决什么问题？达到什么效果？
- 涉及哪些现有模块/文件？新增还是改造？
- 有什么技术偏好或限制？（语言、框架、兼容性）
- 完成后怎么验证？

**退出条件**：上述问题均得到明确回答 → 输出方案方向摘要 → 进入确认步骤。

> 区别于 SOP 阶段一：轻量立项不产出 `docs/proposal.md`，不执行 T-01~T-05 技术分析。
> 讨论结果直接进入 ② 定方案。

### ② 定方案

- 按下方模板写入 `docs/<方案名>.md`
- 更新 `status.md`「轻量方案」字段为 `📝 docs/<方案名>.md — <方案标题>`
- 更新本文件底部「记录」表（Agent 执行时自动填写）
- 方案写入完成后 Agent 提示：「方案已就绪，是否开始执行？（y/n）」
- 用户确认 `y` → 进入 ③；`n` → 方案保持 📝 活跃，等待后续触发

### ③ 执行

- 按方案中的「执行步骤」逐项实施
- 每完成一步，勾选对应的 `- [ ]` → `- [x]`
- 当前 agent 或后续 agent 接力均可，不固定会话边界
- 执行中发现问题及时更新方案文档的风险/变更清单章节
- 执行步骤遇到阻塞（依赖缺失、环境问题等）→ 通知用户，等待决策（继续/跳过/搁置）

**完成标准**：验收标准全部通过 + 执行步骤全部勾选 `[x]` → 视为执行完毕，可进入归档。

### ④ 标注 + 归档

**触发**：Agent 在验收标准和执行步骤均确认完成后，提示用户：
```
方案「<方案名>」执行完毕，是否归档？（y/n）
```
用户确认 `y` 后执行以下归档动作：

- 修改方案文档的「执行状态」为 `✅ 已落地（YYYY-MM-DD）`
- 已完成的 `- [ ]` 全部勾选为 `- [x]`
- 更新 `status.md` 记录完成，清回「轻量方案: [无]」
- 文件从 `docs/` 移入 `docs/archived/`
- 如 `docs/archived/` 不存在，Agent 先创建再移动
- 更新本文件底部「记录」表状态为 `✅`

用户确认 `n` → 方案保持 `📝 活跃` 状态，留在 `docs/`，等待下次会话继续。

### ⏸ 搁置（中途放弃）

用户在任何阶段明确表示"不做了"或"搁置"时：
- 修改「执行状态」为 `⏸ 搁置`
- 保留已完成的 `[x]` 勾选状态
- 更新 `status.md`「轻量方案」字段为 `[无]`
- 文件移入 `docs/archived/`（不存在则创建）
- 更新本文件底部「记录」表状态为 `⏸`
- 无需经过 y/n 确认（用户已直接表态）

## 方案文档模板

> 此模板用于 `docs/<方案名>.md`（方案文档），状态标记含义见下方「状态标记说明」。
> 与本 workflow 文件自身的 `📄 活跃引用文档` 标记不同：`📄` 用于 workflow/参考文档，`📝`/`✅`/`⏸`/`📦` 用于方案文档。

```markdown
# <方案标题>

> **执行状态**: 📝 活跃 | YYYY-MM-DD
>
> 一句话描述定位。

---

## 目标

要解决什么问题 / 达到什么效果。

## 方案（含设计决策）

技术方案描述，关键设计决策及其理由。

## 验收标准

- [ ] 标准 1：...
- [ ] 标准 2：...

## 依赖

- 前置条件 1
- 依赖项 2

## 变更清单

| 文件 | 操作 | 说明 |
|------|------|------|

## 执行步骤

- [ ] Step 1: ...
- [ ] Step 2: ...
- [ ] Step 3: ...

## 风险

| 风险 | 等级 | 应对 |
|------|------|------|
```

## 状态标记说明

| 标记 | 含义 | 位置 |
|------|------|------|
| `📝 活跃` | 方案已定稿，待执行 | `docs/` |
| `⏸ 搁置` | 不做了，保留存档 | `docs/archived/` |
| `✅ 已落地（日期）` | 执行完毕 | `docs/archived/` |
| `📦 参考已归档` | 参考性文档，不涉及执行 | `docs/archived/` |

## 与 SOP 五阶段的区别

| | SOP 五阶段 | 轻量开发 |
|--|-----------|---------|
| 会话边界 | 固定（阶段结束时结束会话） | 不固定 |
| 固定产出 | proposal → design → tasks → prompt | 仅一份 docs/<方案名>.md |
| 触发方式 | 口令（阶段一~五） | `立项` 口令 |
| 归档要求 | 阶段完成 → 存档指令 | 执行完毕 → 更新标记 + 移入 archived |
| 适用场景 | 大型功能、需多 agent 接力 | 中小型功能、技术改造 |
