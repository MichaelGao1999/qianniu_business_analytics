# 恢复指令 — 完整执行流程

> **执行状态**: 📄 通用引用文档 — 被 AGENTS.md §4.2 调用
>
> 本文件实现恢复流程，采用**线性步序 + STOP 门禁**模式。**必须逐行按顺序执行，不可跳步、合并、重排。每步完成后输出 `[步骤 N ✅]` 标记方可进入下一步。**
>
> 适用于所有配置了 `AGENTS.md` 的项目。部分步骤（记忆系统）以条件块标注，无对应工具时自动跳过。

---

## 前置规则（不可违反）

| 规则 | 内容 |
|------|------|
| **线性步序** | 步0→1→2→3→4→5→6 严格执行，每步完成后方可进入下一步 |
| **只读原则** | 同步操作仅限于 `git fetch && git pull --ff-only`，不做任何工作区写入之外的自动操作 |
| **脏即停** | 工作区有未提交修改时，不自动处理（不 stash、不 commit），提示用户先手动方案 |
| **记忆同步做** | 步4 的记忆 pull 和 search 不得跳步（如可用） |

---

## 管线总览

| 步 | 阶段 | 动作 | 产出物 | Exit Condition | Decision Type | 用户交互 |
|----|------|------|--------|---------------|---------------|---------|

| 1 | 工作区检查 | `git status --porcelain` | 工作区状态报告 | 干净或已通知用户 | Deterministic | ⚠️ 脏则停 |

## 步 0 — 分支确认

**目标**：确认当前所在分支为 `main`，否则阻止恢复流程。

> **STOP.** 执行前确认当前工作目录是项目根目录。

```bash
git branch --show-current
```

- **输出为 `main`** → ✅ 继续
- **输出为其他分支** → ❌ 停止，输出提示：
  > 当前不在 main 分支。请先切换（`git checkout main`）后再执行恢复。

**产出物**：分支名称确认。

**Exit Condition**：确认当前在 main 分支。否则停止。

---

## 步 1 — 工作区状态检查

**目标**：确认工作区无未提交修改，确保 `git pull --ff-only` 干净执行。

> **STOP.** 步0 的 `main` 分支确认必须已完成。

```bash
git status --porcelain
```

- **无输出（工作区干净）** → ✅ 继续，进入步2
- **有输出（工作区脏）** → ❌ 停止，输出提示：
  > 工作区有未提交修改：
  > ```
  > {git status --short 的输出}
  > ```
  > 恢复流程要求干净工作区。请确认：
  > - 想保留这些改动 → 先提交或 stash，再重新执行恢复
  > - 想丢弃这些改动 → 手动 `git stash` 后再恢复
  >
  > 恢复已中止，等待你手动处理后重新执行。

**不自动处理脏状态。** 不 stash、不 commit、不 merge。用户必须手动解决。

**产出物**：工作区状态确认（干净或中止通知）。

**Exit Condition**：工作区干净（`git status --porcelain` 无输出）。脏则中止。

---

## 步 2 — Git 同步

**目标**：拉取远端 latest commit，以 `--ff-only` 安全对齐。

> **STOP.** 步1 的工作区干净确认必须已完成。脏工作区禁止执行此步。

```bash
git fetch origin
```

`git fetch` 无副作用，仅拉取远端索引。

```bash
git pull --ff-only
```

`--ff-only` 保证：
- 本地可快进 → 自动完成对齐
- 本地超前于远端 → 不做任何事（本地修改安全保留）
- 本地与远端分叉 → **拒绝执行**（不会产生 merge commit，不会产生冲突标记）

**如果 `git pull --ff-only` 失败**（本地与远端分叉），输出提示：
> 本地与远端 main 分支已分叉，无法快进对齐。请手动处理后再恢复：
> - 如本地改动重要 → 先提交再 `git reset --hard origin/main` 后重新恢复
> - 如远端改动重要 → `git pull --rebase` 手工合并

**产出物**：本地 HEAD 已对齐到 `origin/main`（或失败提示）。

**Exit Condition**：`git pull --ff-only` 成功，或失败后已通知用户中止。

---

## 步 3 — 读取状态文件

**目标**：获取项目当前进度和会话历史。

> **STOP.** 步2 的 Git 同步必须已完成（或确认无远端更新）。

### 3a. 读取 status.md（主数据源）

读取 `status.md`，获取：
- 当前阶段、进度总览
- 待办清单（未完成项）
- 技术债务
- 环境备忘
- 更新记录（最近生效变更）

### 3b. 读取 session-log.md（辅数据源）

读取 `session-log.md` **仅最后一条记录**，提取：
- 遗留问题 / 下轮开始点

禁止复述整条 session-log 历史。`status.md` 为主，session-log 只用于验证一致性。

**产出物**：当前状态的结构化摘要。

**Exit Condition**：两份文件均已读取。

---

## 步 4 — 记忆同步（如可用）

**目标**：从持久化存储加载相关上下文。

> **STOP.** 步3 的状态文件必须已读取。
>
> 如项目未配置记忆系统（无 `awb CLI`、`.memory/` 目录或等效工具），跳过此步。

```bash
if command -v awb &>/dev/null && [ -d .memory ]; then
    awb memory pull     # 从 .md 文件重建本地 memory.db
    awb memory search   # 查找与当前项目相关的记忆上下文
else
    echo "无记忆系统，跳过步4"
fi
```

**产出物**：记忆上下文已加载到工作记忆中（如可用）。

**Exit Condition**：搜索完成（或跳过条件满足）。

---

## 步 5 — 分析 + 汇报

**目标**：综合所有信息输出恢复摘要。

> **STOP.** 步4（如执行）的记忆搜索必须已完成。

1. **检查报错**：用户当前有报错 → 搜索 `troubleshooting.md` 或 `experience-index.md`（如该文件存在）
2. **载入经验知识**：从 `status.md` 待办/阶段描述中提取关键词（如技术栈、模块名），`grep` 或 `awb search` 搜索 `experience-index.md` 匹配前 5 条，读入工作记忆但不展示。如经验文件不存在则跳过。
3. **检查状态有效性**：`status.md` 关键字段是否为占位符（`[填写]`、`[项目名]`）、`session-log.md` 是否无记录
4. **综合判断**：当前阶段、进度、阻塞项、遗留问题，结合已载入的经验知识做判断
5. **输出恢复摘要**

### 恢复摘要模板

```
【恢复完成】

📌 当前：阶段X — [阶段名]
✅ 已完成：[关键里程碑]
📋 待办：[取前 3 条]
🔴 阻塞：[如有]
📁 关键文件：[3-5 个核心入口文件]
🚀 建议下一步：[根据当前阶段给出建议]
```

**产出物**：恢复摘要已输出。

**Exit Condition**：摘要输出完成。

---

## 步 6 — 等待用户指令

**目标**：呈现恢复结果，等待用户决定方向。

> **STOP.** 步5 的恢复摘要必须已输出。

等待用户输入下一步指令。不做任何自动推进。

**产出物**：无。

**Exit Condition**：用户已给出指令。

## 末尾 Self-check（必做）

恢复完成后，逐项核验：

- [ ] 步0 — 分支确认：当前在 `main`
- [ ] 步1 — 工作区干净确认（脏则中止，非自动处理）
- [ ] 步2 — `git pull --ff-only` 成功（或失败已通知）
- [ ] 步3 — status.md + session-log.md 已读取
- [ ] 步4 — 记忆同步已完成（或跳过条件满足）
- [ ] 步5 — 恢复摘要已输出
- [ ] 步6 — 等待用户指令
- [ ] 步序 — 步0→1→2→3→4→5→6 线性执行，未跳步

**任一项未勾选 → 回到对应步骤补做。**
