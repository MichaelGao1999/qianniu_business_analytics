## 决策清单

### B-01：我们只服务淘系生意参谋，不做全渠道

- **我们定了什么**：这个技能只处理天猫/淘宝生意参谋导出的 Excel。用户提到京东、抖音、拼多多、全渠道、多渠道，一律明确拒绝。
- **为什么定**：不同平台的数据口径不一样（比如 UV 怎么算、支付金额包不包含退款），强行合并会误导用户。其他平台用其他技能处理。
- **不能改什么**：`analyze_excel_report.py` 不解析非生意参谋格式的 Excel。遇到非淘系诉求必须明确拒绝。

### B-02：Token Key 只问用户一次 — ~~已废弃（Excel 模式）~~

- ~~旧规则：Cookie 过期系统自动刷新，不重复向用户索要 Token Key。~~
- **2026-06-25 更新**：Excel 驱动模式无需任何认证凭证。`auth/jycm.json` 和 API 取数流程已全部删除。用户只需将生意参谋导出的 Excel 放入 `data/` 目录。

### B-03：结束日期用零点，不用 23:59:59（继续有效）

- **我们定了什么**：所有 `endDate` 统一写成 `T00:00:00+08:00`（结束日当天零点），禁止使用 `T23:59:59.999+08:00` 和 `Z`（UTC）后缀。
- **为什么定**：后端实测，`23:59:59` 会导致多返回一天数据。`Z` 后缀会被后端当成次日早上 7 点多，也会多一天。
- **不能改什么**：日期时间格式强制 `YYYY-MM-DDT00:00:00+08:00`，这是和后端的约定，动不了。
- **当前状态**：Excel 模式下日期格式主要由 `constants.py` 中的 `DATE_FMT` 和 `ISO8601_BJ_START` 管理，用于报告标题和文件命名。

- ~~旧规则：`auth/jycm.json` 必须存三个字段，刷新只改 Cookie 字段。~~
- **2026-06-25 更新**：All deleted. Excel 驱动模式不涉及 Cookie 管理。

### B-05：单店和多店用同一套报告结构（继续有效）

- **我们定了什么**：报告统一四段式（整体指标 / 日度趋势 / 店铺排行 TOP / 分析总结），单店和多店都用这个结构。
- **为什么定**：单店和多店的差异只在"排行段"和"总结段"怎么展开，骨架一样。分开两套模板维护成本高。
- **不能改什么**：四段式结构不能变。单店排行用商品/渠道维度，多店排行用店铺维度。

### B-06：对话交付和钉钉推送用同一套 Markdown（继续有效）

- **我们定了什么**：先在对话里把 Markdown 报告定稿，然后原样推送到钉钉。禁止对话里一版、钉钉里另写一版。
- **为什么定**：避免信息不一致，减少维护成本。钉钉原生支持 Markdown。
- **不能改什么**：推送正文必须与对话中定稿的 Markdown 逐字一致（超长截断除外）。

### B-07：店铺 ID 必须是字符串数组 — ~~已废弃（Excel 模式）~~

- ~~旧规则：`shopIds` 传给后端时必须是 `List<String>`，不能传数字数组。~~
- **2026-06-25 更新**：Excel 模式不涉及 API shopIds 参数。该知识保留在 `lessons-learned.md` 中供参考。

### B-08：预检结果一次性输出，不拆多轮（继续有效）

- **我们定了什么**：所有检查项的结果必须在**同一条回复**里一次性给出。
- **为什么定**：拆成多轮对话会让用户焦虑，也浪费对话轮次。
- **不能改什么**：禁止先问"订没订"、再问"绑没绑"、最后要请求码的三轮拆分。
- **当前状态**：Excel 模式下预检简化为检查 `data/` 目录是否有 `.xlsx` 文件。
