## 决策清单

### B-01：我们只服务淘系生意参谋，不做全渠道

- **我们定了什么**：这个技能只处理天猫/淘宝的生意参谋数据。用户提到京东、抖音、拼多多、全渠道、多渠道，一律明确拒绝。
- **为什么定**：不同平台的数据口径不一样（比如 UV 怎么算、支付金额包不包含退款），强行合并会误导用户。其他平台用其他技能处理。
- **不能改什么**：`channelName` 固定为 `天猫淘宝`，`dataPlatform` 固定为 `生意参谋`。代码里遇到非淘系诉求必须终止执行。

### B-02：Token Key 只问用户一次

- **我们定了什么**：用户第一次使用时给一个 Token Key（请求码），之后 Cookie 过期了由系统自动刷新，再也不问用户要。
- **为什么定**：反复问用户要密码/验证码体验极差。系统用 AK/SK 走 Digest 认证自己换 Cookie。
- **不能改什么**：除非 `auth/jycm.json` 文件不存在，或者 AK/SK 被服务端明确拒绝，否则**禁止**向用户索要 Token Key。

### B-03：结束日期用零点，不用 23:59:59

- **我们定了什么**：所有 `endDate` 统一写成 `T00:00:00+08:00`（结束日当天零点），禁止使用 `T23:59:59.999+08:00` 和 `Z`（UTC）后缀。
- **为什么定**：后端实测，`23:59:59` 会导致多返回一天数据。`Z` 后缀会被后端当成次日早上 7 点多，也会多一天。
- **不能改什么**：日期时间格式强制 `YYYY-MM-DDT00:00:00+08:00`，这是和后端的约定，动不了。

### B-04：AK/SK 和 Cookie 一起存，刷新时只更新 Cookie

- **我们定了什么**：`auth/jycm.json` 必须存三个字段（accessKey、secretKey、jycmOpenApiCookie）。Digest 刷新拿到新 requestCode 后，只改 `jycmOpenApiCookie` 字段，AK/SK 不动。
- **为什么定**：如果只存 Cookie 不存 AK/SK，Cookie 过期后就再也没法刷新了，只能重新找用户要 Token Key。
- **不能改什么**：初始化时必须三字段完整写入；刷新时禁止覆盖整个文件导致 AK/SK 丢失。

### B-05：单店和多店用同一套报告结构

- **我们定了什么**：报告统一四段式（整体指标 / 日度趋势 / 店铺排行 TOP / 分析总结），单店和多店都用这个结构。
- **为什么定**：单店和多店的差异只在"排行段"和"总结段"怎么展开，骨架一样。分开两套模板维护成本高。
- **不能改什么**：四段式结构不能变。单店排行用商品/渠道维度，多店排行用店铺维度。

### B-06：对话交付和钉钉推送用同一套 Markdown

- **我们定了什么**：先在对话里把 Markdown 报告定稿，然后原样推送到钉钉。禁止对话里一版、钉钉里另写一版。
- **为什么定**：避免信息不一致，减少维护成本。钉钉原生支持 Markdown。
- **不能改什么**：推送正文必须与对话中定稿的 Markdown 逐字一致（超长截断除外）。

### B-07：店铺 ID 必须是字符串数组

- **我们定了什么**：`shopIds` 传给后端时必须是 `List<String>`（JSON 字符串数组），不能传数字数组。
- **为什么定**：后端接口要求字符串类型，数字数组会导致参数校验失败。
- **不能改什么**：所有 `shopIds` 的构造必须确保元素为 `str` 类型。

### B-08：预检结果一次性输出，不拆多轮

- **我们定了什么**：不管是首次初始化还是后续预检，所有检查项（授权/订购/绑店）的结果必须在**同一条回复**里一次性给出。
- **为什么定**：拆成多轮对话会让用户焦虑，也浪费对话轮次。
- **不能改什么**：禁止先问"订没订"、再问"绑没绑"、最后要请求码的三轮拆分。
